## Code Audit Suggestions

I used the created file `plan-codebaseAudit.prompt.md` to guide a focused review of the repo and identified concrete maintainability and duplication issues across the main process, renderer, and widgets.

### Key Findings

1. Main process / IPC layer

- `src/main/ipc.ts`
  - Lots of repeated validation wrapper patterns:
    - `kvHandler`
    - `sqlQueryHandler`
    - `secretsHandler`
    - `googleServiceHandler`
  - Repeated manual `ipcMain.handle` registration and argument type checks.
  - `SHELL`, `NET`, `DIALOG` handlers also duplicate error-handling style and validation.
- `src/main/storage/json.ts`
  - `widgetDir`, `file`, `load`, `schedule`, and `flush` are fairly clean, but the file path and atomic write logic is duplicated in `src/main/storage/index.ts`.
  - `flushAll` and `invalidateCache` are separate but could share path resolution helpers.
- `src/main/storage/sqlite.ts`
  - `dbFor` creates directories synchronously on the main thread.
  - `run`, `all`, `get`, `exec`, and `runBatch` all prepare statements and call `dbFor` repeatedly; there is duplication around statement execution and `onWritten` invocation.
  - The `runBatch` transaction logic is good, but the boilerplate can be centralized.

2. Renderer widget infrastructure

- `src/renderer/src/plugins/api.ts`
  - The wrapper around `window.cc` is explicit but repetitive.
  - `kv`, `sql`, `secrets`, and `google` all follow the same pattern and could be generated by small factory functions.
  - `google.shared.reconnect` contains widget-specific logic that could be extracted into a shared helper or moved to a shared service utility.
- `src/renderer/src/plugins/registry.ts`
  - Widget registration performs repeated validation and platform filtering logic.
  - `defaultSettingsFor` manually loops over fields; a small reusable helper could reduce boilerplate.
- `src/renderer/src/state/dashboard.ts`
  - Repeated active-dashboard patching across `removeInstance`, `updateLayout`, `updateSettings`, and `setTitle`.
  - Similar `map`/`patchActive` patterns could be extracted into a generic `updateInstance` helper.
  - `mutate` + `persist` is a good pattern, but the normalization/debugging flow is duplicated in `load` and `applyRemoteState`.
- `src/renderer/src/components/WidgetSettingsPanel.tsx`
  - The field-rendering `switch` is already reasonable, but the component is a bit large and could be simplified by extracting each field type into its own renderer or component.
  - `draft` state is stored as an untyped record, which makes this area more error-prone as new setting kinds are added.
- `src/renderer/src/components/Dashboard.tsx`
  - `layout` and `gridItems` both iterate `dashboard.instances` and look up widgets separately.
  - Repeated `getWidget(i.widgetId)` calls could be memoized once.
  - The current implementation mixes layout generation and DOM query-based scrolling logic; separating these concerns would improve readability.

3. Widget code / UI styling

- `src/widgets/job-tracker`
  - Repeated suggestion/autocomplete logic for company/role/location fields.
  - `handleImportCSV` duplicates the insert SQL parameter list and normalization logic.
  - Many inline styles are used throughout widget code, making it harder to maintain and reuse.
- Shared widget utilities
  - Only one shared component exists under `src/widgets/_shared/StackedBarChart.tsx`.
  - There is an opportunity to create reusable widget helpers for:
    - common buttons
    - form field wrappers
    - suggestion/autocomplete logic
    - list item cards
    - shared style tokens
- General widget pattern
  - Multiple widgets contain large style objects and inline styles, which increases maintenance cost.
  - A small shared UI/style layer would reduce duplication and make widget authoring easier.

### Recommendations

#### High-priority refactors

1. Extract generic IPC handler helpers in `src/main/ipc.ts`
   - e.g. `registerHandler(channel, validator, handler)`
   - e.g. `withArgs(channel, schema, fn)`
2. Extract shared storage path and atomic-write helpers
   - unify `JsonStore.flush` and `Storage.saveState` atomic write logic
   - share widget directory resolution between `JsonStore` and `SqliteStore`
3. Add a renderer state helper in `dashboard.ts`
   - generic `updateWidgetInstance(instanceId, updater)`
   - reduce repeated `map`/`patchActive` code
4. Factor `createWidgetApi` into smaller builders
   - `createKvApi(widgetId, instanceId)`
   - `createSqlApi(widgetId)`
   - `createGoogleApi(widgetId)`
5. Extract settings field renderers in `WidgetSettingsPanel.tsx`
   - separate `SettingsFieldString`, `SettingsFieldNumber`, etc.
   - better typing for `WidgetSettings`

### Execution / maintainability impact

#### What is good

- The repo already has a strong separation between main/preload/renderer/widgets.
- Widget API surface is clearly defined.
- Dashboard state persistence and sync are encapsulated cleanly.
- `SqliteStore` transaction handling is well designed.

#### What is worth improving

- Most duplication is in boilerplate validation, repeated `map`/`filter` state transforms, and inline widget styling.
- Refactoring these areas will improve readability without changing architecture.
- The biggest maintainability win is shared helper extraction, not a rewrite.

### Suggested next step

Start by refactoring the main IPC layer and storage helpers, then move to renderer state helpers and widget UI reuse.
